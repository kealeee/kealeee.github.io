Classic RAG System, Fully Local

GitHub ↗

A Retrieval-Augmented Generation system that answers questions over your documents with no cloud LLM required, and shows its reasoning.

Ingest→ Chunk (LangChain)→ Embed → ChromaDB→ Query rewrite→ Rerank (Cohere)→ Generate (flan-t5)→ Decision trace

The problem

Most RAG demos lean on paid cloud LLMs and behave like black boxes: when an answer is wrong, you cannot see whether retrieval, ranking, or generation failed.

What I built

A complete local pipeline: document ingestion and chunking with LangChain, vector search on ChromaDB, query rewriting to improve recall, Cohere reranking for precision, and answer generation with HuggingFace flan-t5 running locally. Served through a FastAPI backend with a single-page UI.

What makes it different

A dedicated decision-trace tab exposes every stage: which chunks were retrieved, how the query was rewritten, how reranking reordered candidates, and what context reached the generator. Explainability is a feature, not an afterthought — a habit from two decades of diagnosing systems that had to tell you why they failed.

Bedtime Story Generator, Deployed Fullstack

GitHub ↗

A live, production-deployed AI app that generates personalised children's stories, spanning three cloud services.

Frontend (Vercel)→ FastAPI (Render)→ Gemini API→ PostgreSQL (managed)

The problem

A notebook demo is not a product. I wanted the full journey: an AI feature that real users can reach at a URL, with persistence, across independently deployed services.

What I built

Google Gemini generates stories personalised by the child's name, age, and theme through structured prompt engineering. A FastAPI backend on Render handles generation and persistence to a managed PostgreSQL database; a static frontend on Vercel keeps the user experience fast and hosting free.

What makes it different

It exercises the unglamorous parts most AI demos skip: CORS between separately hosted frontend and backend, environment and secret management across two platforms, cold-start behaviour on free-tier hosting, and a database schema for generated content. Deployment is part of the engineering, not an afterthought.

Olist E-Commerce Data Pipeline

GitHub ↗

A production-grade ELT pipeline over the Brazilian Olist e-commerce dataset, built collaboratively with automated quality gates.

Ingest → PostgreSQL→ Transform (dbt)→ Validate (Great Expectations)→ CI/CD (GitHub Actions)

The problem

Raw e-commerce data across many linked tables (orders, items, payments, reviews, geolocation) is unusable for analytics until it is ingested, modelled, and, critically, trusted.

My contribution

Within the team I owned data ingestion into PostgreSQL, transformation models in dbt, and data quality validation with Great Expectations, wired into a GitHub Actions CI/CD workflow so every change is tested before it lands.

What makes it different

Quality is enforced, not hoped for: expectations run automatically in CI, so schema drift and bad data fail the build instead of reaching the warehouse. The same instinct that drove my automated smoke-testing framework in automotive production applies unchanged to data engineering.
